# 贡献指南

本指南旨在节省双方的时间。

## 唯一的规则

**你必须理解你的代码。** 如果你无法解释你的更改做了什么以及它们如何与系统的其他部分交互，你的 PR 将被关闭。

使用 AI 编写代码是可以的。提交未经理解的 AI 生成代码是不可以的。

如果你使用代理，请从 `pi-mono` 根目录运行它，这样它会自动加载 `AGENTS.md`。你的代理必须遵循该文件中的规则和指南。

## 贡献门槛

所有来自新贡献者的 issue 和 PR 默认自动关闭。

周五到周日提交的 issue 不会被审查。如果有紧急事项，请在 Discord 上询问：https://discord.com/invite/3cU7Bz4UPx

维护者每天审查自动关闭的 issue，并重新打开有价值的 issue。不符合以下质量标准的 issue 将不会被重新打开或收到回复。

审批通过维护者在 issue 中的回复进行：

- `lgtmi`：你未来的 issue 将不会被自动关闭
- `lgtm`：你未来的 issue 和 PR 将不会被自动关闭

`lgtmi` 不授予提交 PR 的权利。只有 `lgtm` 才授予提交 PR 的权利。

## Issue 质量标准

如果你打开一个 issue，你必须使用两个 GitHub issue 模板之一。

如果你打开一个 issue，请保持简短、具体且值得阅读。

- 保持简洁。如果一屏放不下，就太长了。
- 用你自己的语言表达。
- 清楚地说明 bug 或请求。
- 解释为什么它很重要。
- 如果你想自己实现这个更改，请说明。

如果 issue 是真实的且写得好，维护者可能会重新打开它，回复 `lgtmi` 或回复 `lgtm`。

## 封禁

如果你两次无视本文档，或者你用代理生成的 issue 轰炸跟踪器，你的 GitHub 账户将被永久封禁。

如果你通过自动化发送大量 issue，你的 GitHub 账户将被永久封禁。没有反悔的机会。

## 提交 PR 之前

除非你已经获得 `lgtm` 批准，否则不要打开 PR。

提交 PR 之前：

```bash
npm run check
./test.sh
```

两者都必须通过。

不要编辑 `CHANGELOG.md`。变更日志条目由维护者添加。

如果你要为 `packages/ai` 添加新的供应商，请参见 `AGENTS.md` 中的必需测试。

## 理念

pi 的核心是极简的。如果你的功能不属于核心，它应该是一个扩展。膨胀核心的 PR 可能会被拒绝。

## 常见问题

### 为什么新的 issue 和 PR 会自动关闭？

pi 收到的 issue 数量超出维护者实时审查的能力。许多报告不符合本指南中的质量标准或不遵循 CONTRIBUTING.md。有些是通过代理不经审查就投向仓库的。自动关闭创建了一个缓冲区，让维护者可以按自己的节奏审查跟踪器，并重新打开符合质量标准的 issue。

### 为什么周末的 issue 不被审查？

维护者需要不受打扰的远离跟踪器的时间。周五到周日提交的 issue 自动关闭，不列入周一审查队列。如果问题紧急，请在 Discord 上询问，包含简短版本、复现步骤和相关日志。

### 为什么有些 issue 没有回复？

回复也是维护工作。低信号的 issue、不清楚的报告、重复项以及不遵循本指南的 issue 可能会不经讨论直接关闭。这样可以将时间留给可复现的 bug、深思熟虑的请求和已经做了工作使其报告可操作的贡献者。

### 为什么不让 AI 做所有的分类？

AI 可以帮助对重复项分组、总结报告和发现缺失信息。它不被信任做最终的维护者决定。经过润色的 AI 生成 issue 仍然可能是错误的、误导性的或调查成本高昂的。人工审查仍然是最终关口。

### 这是否对贡献者不友好？

不是。这是对倦怠和跟踪器垃圾的防护。简短、具体、可复现的 issue 是欢迎的。深思熟虑的贡献是欢迎的。自动化的垃圾、无理要求和大量低质量报告是不欢迎的。
