# 开发规则

## 对话风格

- 保持回答简短和简洁
- 在提交、issue、PR 评论或代码中不使用表情符号
- 不要有客套话或欢快的填充文字（例如，用 "Thanks @user" 而不是 "Thanks so much @user!"）
- 仅使用技术性语言，直截了当
- 当用户提问时，先回答问题，然后再编辑或运行实现命令
- 当回应用户反馈或分析时，在说你做了什么更改之前，明确表示你同意还是不同意

## 代码质量

- 在进行大范围更改之前、在编辑未完全检查的文件之前以及被要求调查或审计时，完整阅读文件。不要依赖搜索片段来进行大范围更改。
- 除非绝对必要，不使用 `any`。
- 内联只有一个调用点的单行辅助函数。
- 检查 node_modules 中的外部 API 类型；不要猜测。
- **禁止内联导入**（`await import()`、`import("pkg").Type`、动态类型导入）。仅使用顶层导入。
- 永远不要为了修复过时依赖的类型错误而删除或降级代码；应该升级依赖。
- 在根配置检查的代码中（`packages/*/src`、`packages/*/test`、`packages/coding-agent/examples`）仅使用可擦除的 TypeScript 语法（Node strip-only 模式）：不使用参数属性、`enum`、`namespace`/`module`、`import =`、`export =` 或其他需要 JS 发射的构造。使用显式字段和构造函数赋值。
- 在删除看起来是有意的功能或代码之前总是先询问。
- 除非用户要求，不保持向后兼容性。
- 永远不要硬编码按键检查（例如 `matchesKey(keyData, "ctrl+x")`）。将默认值添加到 `DEFAULT_EDITOR_KEYBINDINGS` 或 `DEFAULT_APP_KEYBINDINGS` 中，使其保持可配置。
- 永远不要直接修改 `packages/ai/src/models.generated.ts`；而是更新 `packages/ai/scripts/generate-models.ts`，然后重新生成。包含生成的 `models.generated.ts` diff 始终是可以的，即使重新生成包含不相关的上游模型元数据更改。

## 命令

- 代码更改后（非文档）：`npm run check`（完整输出，不截断）。在提交之前修复所有错误、警告和信息。不运行测试。
- 除非用户请求，永远不要运行 `npm run build` 或 `npm test`。
- 永远不要直接运行完整的 vitest 套件：当端点/认证环境变量存在时，它包含 e2e 测试。对于所有非 e2e 测试，从仓库根目录运行 `./test.sh`。否则从包根目录运行特定测试：`node ../../node_modules/vitest/dist/cli.js --run test/specific.test.ts`。
- 如果你创建或修改了测试文件，运行它并在测试或实现上迭代直到通过。
- 对于 `packages/coding-agent/test/suite/`，使用 `test/suite/harness.ts` + faux provider。不使用真实的供应商 API、密钥或付费令牌。
- 将 issue 特定的回归测试放在 `packages/coding-agent/test/suite/regressions/` 下，命名为 `<issue-number>-<short-slug>.test.ts`。
- 对于临时脚本，`write` 到临时文件（例如 `/tmp`），运行，如需编辑，完成后删除。不要在 `bash` 命令中嵌入多行脚本。
- 除非用户要求，不要提交。

## 依赖和安装安全

- 将 npm 依赖和 lockfile 更改视为已审查的代码。直接外部依赖固定为精确版本。
- 本地安装/更新使用 `npm install --ignore-scripts`；干净/CI 风格使用 `npm ci --ignore-scripts`。除非用户要求，不运行生命周期脚本。
- 如果依赖元数据更改，使用 `npm install --package-lock-only --ignore-scripts` 刷新 `package-lock.json`。
- 如果 `packages/coding-agent/npm-shrinkwrap.json` 需要重新生成，运行 `node scripts/generate-coding-agent-shrinkwrap.mjs`（使用 `--check` 或 `npm run check` 验证）。带有生命周期脚本的新依赖需要审查和该脚本中的显式允许列表条目；永远不要静默添加。
- 预提交钩子会阻止 lockfile 提交，除非设置 `PI_ALLOW_LOCKFILE_CHANGE=1`。除非用户要求提交 lockfile 更改，否则不要绕过。

## Git

多个 pi 会话可能同时在此工作目录中运行，每个修改不同的文件。触及你自己更改之外的未暂存、已暂存或未跟踪文件的 Git 操作将破坏其他会话的工作。遵循以下规则：

提交：

- 只提交你在本次会话中更改的文件。
- 暂存明确的路径（`git add <path1> <path2>`）；永远不要 `git add -A` / `git add .`。
- 提交前运行 `git status` 并验证你只暂存了你的文件。
- `packages/ai/src/models.generated.ts` 可以始终与你的文件一起包含。

永远不要运行（破坏其他代理的工作或绕过检查）：

- `git reset --hard`、`git checkout .`、`git clean -fd`、`git stash`、`git add -A`、`git add .`、`git commit --no-verify`。

如果发生 rebase 冲突：

- 只解决你修改的文件中的冲突。
- 如果冲突在你未修改的文件中，中止并询问用户。
- 永远不要强制推送。

## Issue 和 PR

参见 `CONTRIBUTING.md` 了解贡献者门槛（自动关闭工作流、`lgtm`/`lgtmi`、质量标准）。

审查 PR 时：

- 不要运行 `gh pr checkout`、`git switch` 或以其他方式将工作树移动到 PR 分支，除非用户明确要求。
- 使用 `gh pr view`、`gh pr diff`、`gh api` 和本地 `git show`/`git diff` 对获取的引用进行检查 PR 元数据、提交和补丁，而不改变分支。
- 如果你需要 PR 文件内容，将它们获取/读取到临时文件或使用 `git show <ref>:<path>` 而不切换分支。

创建 issue 时：

- 为受影响的包添加 `pkg:*` 标签（`pkg:agent`、`pkg:ai`、`pkg:coding-agent`、`pkg:tui`）；使用所有适用的标签。

发布 issue/PR 评论时：

- 将评论写入临时文件，并使用 `gh issue/pr comment --body-file` 发布（永远不要通过 `--body` 传递多行 markdown）。
- 保持评论简洁、技术性，符合用户的语气。
- 在每个 AI 发布的评论末尾添加发起提示指定的 AI 生成免责声明行（例如 `This comment is AI-generated by /wr`）。

通过提交关闭 issue 时：

- 在消息中包含 `fixes #<number>` 或 `closes #<number>`，这样合并时会自动关闭 issue。对于多个 issue，每个 issue 重复关键字（`closes #1, closes #2`）；共享关键字（`closes #1, #2`）只关闭第一个。

## 使用 tmux 测试 pi 交互模式

在受控终端中运行 TUI（从仓库根目录）：

```bash
tmux new-session -d -s pi-test -x 80 -y 24
tmux send-keys -t pi-test "./pi-test.sh" Enter
sleep 3 && tmux capture-pane -t pi-test -p     # 启动后捕获
tmux send-keys -t pi-test "your prompt here" Enter
tmux send-keys -t pi-test Escape               # 特殊键（也可以用 C-o 表示 ctrl+o 等）
tmux kill-session -t pi-test
```

## 变更日志

位置：`packages/*/CHANGELOG.md`（每个包一个）。

`## [Unreleased]` 下的章节：`### Breaking Changes`（需要迁移的 API 更改）、`### Added`、`### Changed`、`### Fixed`、`### Removed`。

规则：

- 所有新条目放在 `## [Unreleased]` 下。先阅读完整章节并追加到现有子章节；永远不要重复它们。
- 已发布的版本章节（例如 `## [0.12.2]`）是不可变的；永远不要修改它们。

归属：

- 内部（来自 issue）：`Fixed foo bar ([#123](https://github.com/earendil-works/pi-mono/issues/123))`
- 外部贡献：`Added feature X ([#456](https://github.com/earendil-works/pi-mono/pull/456) by [@username](https://github.com/username))`

## 发布

**锁步版本**：所有包共享一个版本；每次发布一起更新所有包。`patch` = 修复 + 新增，`minor` = 破坏性更改。没有主版本发布。

1. **更新变更日志**：询问用户是否已在 `main` 的最新提交上运行了 `/cl` 提示。如果没有，他们必须先运行 `/cl` 来审计和更新每个包的 `[Unreleased]` 章节，然后再发布。

2. **本地冒烟测试**：构建一个未发布的版本并从仓库外进行冒烟测试（这样它不能解析工作区文件）：
   ```bash
   npm run release:local -- --out /tmp/pi-local-release --force
   cd /tmp

   # Node 包安装冒烟测试
   /tmp/pi-local-release/node/pi --help
   /tmp/pi-local-release/node/pi --version
   /tmp/pi-local-release/node/pi --list-models
   /tmp/pi-local-release/node/pi -p "Say exactly: ok"
   /tmp/pi-local-release/node/pi

   # Bun 二进制冒烟测试
   /tmp/pi-local-release/bun/pi --help
   /tmp/pi-local-release/bun/pi --version
   /tmp/pi-local-release/bun/pi --list-models
   /tmp/pi-local-release/bun/pi -p "Say exactly: ok"
   /tmp/pi-local-release/bun/pi
   ```
   验证 Node 和 Bun 的启动、模型/账户列表、交互式启动以及至少一个使用预期默认供应商的真实提示。裸命令 `/tmp/pi-local-release/node/pi` 和 `/tmp/pi-local-release/bun/pi` 启动交互模式；在 tmux 中运行每个，提交一个提示，并等待模型回复，然后才认为交互式冒烟测试通过。失败是发布阻断，除非用户明确接受风险。

3. **运行发布脚本**：
   ```bash
   PI_ALLOW_LOCKFILE_CHANGE=1 npm_config_min_release_age=0 npm run release:patch    # 修复 + 新增
   PI_ALLOW_LOCKFILE_CHANGE=1 npm_config_min_release_age=0 npm run release:minor    # 破坏性更改
   ```
   仅对发布命令使用 `npm_config_min_release_age=0`。仓库正常的 npm 年龄门控在当前工作区包版本最近发布时可能会阻止发布 lockfile 刷新。在推送之前审查发布创建的任何 lockfile 或 shrinkwrap diff。

   发布脚本会升级所有包版本，更新变更日志，重新生成发布工件，运行 `npm run check`，提交 `Release vX.Y.Z`，打标签 `vX.Y.Z`，添加新的 `## [Unreleased]` 变更日志章节，提交 `Add [Unreleased] section for next cycle`，然后推送 `main` 和标签。标签推送后不要重新运行发布脚本。

4. **CI 发布 npm 包**：推送 `vX.Y.Z` 标签会触发 `.github/workflows/build-binaries.yml`。`publish-npm` 作业通过 GitHub Actions OIDC 使用 npm 可信发布，环境为 `npm-publish`；不需要本地 `npm publish`、`npm whoami`、OTP 或 WebAuthn 流程。

5. **如果 CI 发布失败**：检查失败的 `publish-npm` 作业。发布辅助程序是幂等的，会跳过已在 npm 上存在的包版本，所以在修复 CI 或临时 npm 问题后重新运行标签工作流。不要对同一版本重新运行 `npm run release:patch` 或 `npm run release:minor`。
