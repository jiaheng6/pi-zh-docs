# 持久化 AgentHarness 和会话设计

持久化 AgentHarness / 会话设计笔记。

## 框架

完全持久化的 `AgentHarness` 本身是不现实的，因为重要的依赖是主机应用提供的运行时 JS：

- 工具实现
- 模型/认证供应商
- 扩展和 hook 处理器
- 资源加载器
- 系统提示回调/修改器

实际目标是半持久化 harness：

- 会话是持久化的追加 only 状态树
- harness 将它拥有的状态持久化到会话条目中
- 主机应用负责在恢复时重建兼容的不可持久化依赖
- 恢复从持久化边界重启，而非从进行中的供应商流重启

## 会话拥有持久化状态

将会话视为所有持久化代理状态，不仅仅是对话历史。

现有会话状态已包含 harness 状态：模型更改、思考级别更改、活跃工具更改、叶子条目、标签、压缩和分支摘要、自定义消息和条目。

## 应用在恢复时必须提供什么

应用必须重建兼容的运行时依赖：

- 模型注册表/模型对象
- 工具注册表
- 扩展集、版本和排序
- 资源加载器
- 系统提示供应商/hooks
- 认证供应商
- 应用特定 hooks

## 恢复模型

启动时：

1. 主机应用注册工具/模型/扩展/资源/认证/hooks
2. Harness 打开会话
3. Harness 将会话条目归约为：当前叶子、对话分支、harness 配置（包括活跃工具名称）、队列、待处理写入、活跃操作/回合/工具状态
4. Harness 验证所需的运行时依赖
5. Harness 协调未完成的操作状态

供应商流不可恢复。恢复只能从持久化边界重试或标记操作为中断。

## 恢复策略

默认保守策略：

- 未完成的代理回合：标记为中断，保留持久化队列/待处理写入，返回空闲
- 未完成的供应商请求：标记为中断；不自动重试
- 未完成的工具调用：追加中断/错误工具结果；仅在工具声明 retry-safe/幂等时重试
- 未完成的压缩：如果没有压缩条目则重新运行
- 未完成的分支摘要/树导航：如果安全则重新运行/应用缺失的摘要或叶子条目

## 关键场景

### 队列

- 排队已接受前崩溃：消息未被接受
- 排队已接受后崩溃：消息被恢复
- 队列排空后但持久化回合记录之前崩溃：丢失/重复风险
- 必需不变量：已消费的队列 ID 必须在 `turn_started` 中记录后才被视为已消费

### 待处理写入

- 入队前崩溃：写入未被接受
- 入队后应用前崩溃：恢复时应用
- 应用后标记前崩溃：确定性目标条目 ID 让恢复可以检测条目已存在

### 代理循环回合

- 供应商请求前崩溃：重试或标记中断
- 供应商请求期间崩溃：默认标记中断
- 供应商响应后助手消息持久化前崩溃：响应丢失，除非供应商结果已记录日志
- 助手消息持久化后崩溃：从持久化消息恢复

### 工具调用

- 工具调用开始后结果前崩溃：外部副作用可能已发生
- 默认恢复不应重新运行非幂等工具
- 工具调用需要稳定 ID 和重试安全元数据用于自动恢复

## 最小可行原型

1. 添加持久化队列条目
2. 添加带确定性目标 ID 的持久化待处理写入条目
3. 添加操作开始/完成/中断条目
4. 添加带已消费队列 ID 的回合开始
5. 通过归约会话日志恢复
6. 默认将未完成的代理回合标记为中断
7. 仅在没有最终条目时重新运行未完成的压缩/树操作
8. 除非工具元数据声明 retry-safe，否则不重试未完成的工具调用
